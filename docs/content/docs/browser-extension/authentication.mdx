# Authentication

The extension authenticates with the NetworkNav app using a token-based system. Before the extension can submit captures or fetch tasks, it must be registered with a valid token.

## How Token Auth Works

The authentication flow uses a display token system:

1. An admin generates a token in the app (or inserts one directly in the database)
2. The token has the format `ext_` followed by 32 random bytes encoded as base64url
3. A SHA-256 hash of the token is stored in the `extension_tokens` database table
4. The user enters the full display token in the extension popup
5. The extension sends the display token to `POST /api/extension/register`
6. The server hashes it, looks up the matching row, and returns an `extensionId` and default settings
7. The extension stores the token in `chrome.storage.local` and includes it in all subsequent API requests via the `X-Extension-Token` header

<Callout type="warn">
The full token is stored in the extension's local storage and sent with every API request. It is never persisted on the server after the initial hash is created. If you need to revoke access, mark the token as revoked in the database.
</Callout>

## Generating a Token

<Steps>

### Option A: Admin UI

If the admin panel is set up, navigate to the Admin page in the NetworkNav app and use the token management interface to generate a new extension token. Copy the displayed token before closing the dialog -- it will not be shown again.

### Option B: Manual database insert

If the admin UI is not available, insert a token directly using SQL:

```sql
INSERT INTO extension_tokens (
  extension_id, token_hash, display_prefix, created_at
) VALUES (
  'ext-my-browser-001',
  encode(sha256('ext_your-secret-token-value-here'::bytea), 'hex'),
  'ext_your-sec',
  now()
);
```

Replace `ext_your-secret-token-value-here` with any string you choose. The `display_prefix` is the first 12 characters of the token (e.g., `ext_your-sec`), shown in admin listings for identification without revealing the full token.

If using Docker, run the command via `docker exec`:

```bash
docker exec ctox-db psql -U netnav -d netnav -c "
  INSERT INTO extension_tokens (
    extension_id, token_hash, display_prefix, created_at
  ) VALUES (
    'ext-my-browser-001',
    encode(sha256('ext_my-token-12345'::bytea), 'hex'),
    'ext_my-token',
    now()
  )
  ON CONFLICT DO NOTHING;
"
```

In this example, the display token you will enter in the extension is `ext_my-token-12345`.

</Steps>

## Registering the Extension

<Steps>

### Open the extension popup

Click the extension icon in the Chrome toolbar. Since this is the first launch, you will see the "Connect to App" registration form.

### Enter the display token

Type or paste the full display token (e.g., `ext_my-token-12345`) into the text input field.

### Click Connect

The extension sends the token to `POST /api/extension/register` on the configured app URL (default `http://localhost:3750`).

### Verify registration

If successful, the registration form disappears and is replaced by the main dashboard showing:

- **Captures Today**: 0
- **Queued**: 0
- **Tasks**: 0
- Connection status indicator showing "Connected" (green)

If registration fails, an error message appears: "Registration failed. Check your token and try again."

</Steps>

## Token Storage

After registration, the extension stores two values in `chrome.storage.local`:

| Key | Value | Purpose |
|-----|-------|---------|
| `extensionToken` | The full display token string | Sent as `X-Extension-Token` header on every API request |
| `extensionId` | The server-assigned extension ID (e.g., `ext-my-browser-001`) | Identifies this extension instance for rate limiting and logging |

The server-side `withExtensionAuth()` middleware validates every incoming request by:

1. Reading the `X-Extension-Token` header
2. Computing its SHA-256 hash
3. Looking up the hash in `extension_tokens`
4. Checking `is_revoked` is `false`
5. Validating the request origin (`chrome-extension://` or `localhost` in development)

## Token Security

- The server never stores the raw token. Only the SHA-256 hash is persisted in the database.
- Token validation includes origin checking -- requests must come from a `chrome-extension://` origin or `localhost` (for development).
- The server enforces per-extension rate limiting. Exceeding the rate limit returns HTTP 429 with a `Retry-After` header.
- Failed authentication returns HTTP 401 with a structured JSON error body.

## Re-authentication

You need to re-register the extension when:

- **The token is revoked.** An admin sets `is_revoked = true` on the token row. The extension will receive 401 errors on all API calls.
- **The token is deleted.** The token row is removed from the database.
- **The extension storage is cleared.** If `chrome.storage.local` is wiped (e.g., reinstalling the extension), the stored token is lost.

To re-register, clear the extension's stored data by removing it from `chrome://extensions` and reloading it, then follow the registration steps above with a valid token.

## Verifying Token Status

Check whether a token is active in the database:

```bash
docker exec ctox-db psql -U netnav -d netnav -c "
  SELECT extension_id, display_prefix, is_revoked, last_used_at
  FROM extension_tokens;
"
```

Confirm `is_revoked` is `false` for your token. The `last_used_at` column updates each time the extension makes an authenticated API call.
