# ICP Configuration

The ICP (Ideal Customer Profile) configuration is the core of how contacts are scored, tiered, and searched. Three config files drive the system:

| File | Purpose | Required |
|------|---------|----------|
| `data/icp-config.json` | ICP profiles, scoring weights, tier thresholds, niche keywords | Yes |
| `data/behavioral-config.json` | Behavioral persona definitions and scoring rules | Optional (ships with defaults) |
| `data/referral-config.json` | Referral scoring weights, role tiers, persona thresholds | Optional (ships with defaults) |

## Setup Methods

### Method 1: Conversational via Claude (recommended)

```
/linkedin-prospector set up my ICP config
```

Claude conducts a structured conversation asking about your business, then generates the configuration non-interactively. This is the recommended approach.

### Method 2: Interactive CLI wizard

For humans running scripts directly in a terminal:

```bash
cd .claude/linkedin-prospector/skills/linkedin-prospector
node scripts/configure.mjs wizard
```

### Method 3: Non-interactive CLI

Pass the full configuration as JSON for automation:

```bash
node scripts/configure.mjs generate --json '{...full config JSON...}'
```

### Method 4: Manual edit

Edit `data/icp-config.json` directly, then validate:

```bash
node scripts/configure.mjs validate
```

## The 8-Step Conversational Flow

When you ask Claude to configure your ICP, it follows this structured process:

<Steps>

### Ask about services

> "What services do you offer?"

Each service becomes one ICP profile. Example: "I do AI consulting and fractional CTO work" produces two profiles: `ai-consulting` and `fractional-cto`.

### Ask about target buyers (per service)

> "For [service], who are your ideal buyers? List the job titles starting with highest priority."

Claude maps answers to `rolePatterns`:
- First 3-5 titles become `high` (decision makers with budget authority)
- Next 3-5 titles become `medium` (influencers, recommenders)
- Remaining become `low` (implementers, end users)

Keywords use partial matching: "VP" matches "VP Engineering", "VP Product", etc.

### Ask about target industries

> "What industries are your ideal customers in?"

Mapped to lowercase keywords with synonyms included ("ecommerce" and "e-commerce", "saas" and "software").

### Ask about buying signals

> "What keywords in someone's profile would signal they might need [service]?"

Think about problems ("scaling"), initiatives ("digital transformation"), or technologies ("AI").

### Ask about company size (optional)

> "What's the ideal company size range?"

Defaults to `{ "min": 10, "max": 500 }` if unspecified.

### Ask about search niches

> "What keywords should I search LinkedIn for? Group them into niches."

Each niche is a named group of LinkedIn search keywords.

### Assemble and write

Claude builds the complete config JSON and runs:

```bash
node scripts/configure.mjs generate --json '<assembled JSON>'
```

### Validate and confirm

```bash
node scripts/configure.mjs validate
```

Claude reports back: number of profiles created, number of niches defined, and suggests the next step.

</Steps>

## Config File Reference

### `icp-config.json` Structure

```json
{
  "profiles": { ... },
  "scoring": { ... },
  "goldScore": { ... },
  "tiers": { ... },
  "niches": { ... }
}
```

### Profile Definition

Each profile under `profiles` represents one service or offering:

| Field | Type | Description |
|-------|------|-------------|
| `label` | string | Human-readable name |
| `description` | string | Who this profile targets |
| `rolePatterns.high` | string[] | Decision-maker title keywords (score: 1.0) |
| `rolePatterns.medium` | string[] | Influencer title keywords (score: 0.7) |
| `rolePatterns.low` | string[] | End-user title keywords (score: 0.3) |
| `industries` | string[] | Industry keywords (lowercase, partial match) |
| `signals` | string[] | Buying intent keywords matched against headline/about |
| `companySizeSweet` | object | `{ "min": N, "max": N }` ideal employee count |
| `weight` | number | Profile importance multiplier (0.0-1.0) |

### Scoring Weights

Controls how the four ICP fit dimensions are weighted (must sum to 1.0):

| Field | Default | Description |
|-------|---------|-------------|
| `roleWeight` | 0.35 | How much title match matters |
| `industryWeight` | 0.25 | How much industry match matters |
| `signalWeight` | 0.25 | How much buying signals matter |
| `companySizeWeight` | 0.15 | How much company size fit matters |

### Gold Score Weights

Controls the composite Gold Score used for tier assignment (must sum to 1.0):

| Field | Default | Description |
|-------|---------|-------------|
| `icpWeight` | 0.35 | ICP fit score contribution |
| `networkHubWeight` | 0.30 | Network centrality/hub score |
| `relationshipWeight` | 0.25 | Mutual connections, shared context |
| `signalBoostWeight` | 0.10 | Extra boost from strong signals |

### Tier Thresholds

Contacts are classified into tiers based on their Gold Score (thresholds must be in descending order):

| Tier | Default Threshold | Meaning |
|------|-------------------|---------|
| Gold | 0.55 | High-priority prospects |
| Silver | 0.40 | Worth pursuing |
| Bronze | 0.28 | Keep on radar |
| Watch | < 0.28 | Not a current fit |

### Niche Keywords

Maps niche slugs to LinkedIn search terms:

```json
"niches": {
  "ai": ["AI", "artificial intelligence", "machine learning"],
  "ecommerce": ["ecommerce", "DTC", "Shopify", "e-commerce"]
}
```

## Behavioral Config Reference

Behavioral scoring runs after ICP scoring and identifies network behavior patterns. Most users do not need to modify the defaults.

### Scoring Dimensions

| Dimension | Weight | What It Measures |
|-----------|--------|------------------|
| `connectionPower` | 0.20 | LinkedIn connection count (500+ = max) |
| `connectionRecency` | 0.15 | How recently you connected |
| `aboutSignals` | 0.25 | Behavioral keywords in About section |
| `headlineSignals` | 0.15 | Profile headline patterns |
| `superConnectorIndex` | 0.15 | Combo score: 3+ traits = super-connector |
| `networkAmplifier` | 0.10 | Amplification potential: mutuals x connections |

### Behavioral Personas

| Persona | Criteria |
|---------|----------|
| **Super-connector** | 3+ behavioral traits AND 500+ connections |
| **Content-creator** | Speaker/author/podcast keywords in profile |
| **Silent-influencer** | 500+ connections but low behavioral signals |
| **Rising-connector** | < 500 connections, connected within 6 months |
| **Passive-network** | Default -- no strong behavioral signals |

## Validation

Always validate after modifying config:

```bash
node scripts/configure.mjs validate
```

Checks performed:
- JSON is valid
- `profiles` exists with at least one entry
- Each profile has `rolePatterns`
- `scoring` weights sum to 1.0
- `tiers` are in descending order (gold > silver > bronze)
- Warns if `_example` marker is still present (uncustomized template)

## Example: Solo AI Consultant

```json
{
  "profiles": {
    "ai-consulting": {
      "label": "AI Consulting",
      "description": "Leaders exploring AI for their business",
      "rolePatterns": {
        "high": ["CEO", "CTO", "Founder", "Chief Digital"],
        "medium": ["VP", "Director", "Head of"],
        "low": ["Manager", "Lead"]
      },
      "industries": ["saas", "ecommerce", "fintech", "healthcare"],
      "signals": ["ai", "artificial intelligence", "automation", "digital transformation"],
      "companySizeSweet": { "min": 20, "max": 500 },
      "weight": 1.0
    }
  },
  "scoring": { "roleWeight": 0.35, "industryWeight": 0.25, "signalWeight": 0.25, "companySizeWeight": 0.15 },
  "goldScore": { "icpWeight": 0.35, "networkHubWeight": 0.30, "relationshipWeight": 0.25, "signalBoostWeight": 0.10 },
  "tiers": { "gold": 0.55, "silver": 0.40, "bronze": 0.28 },
  "niches": {
    "ai": ["AI", "artificial intelligence", "machine learning"],
    "saas": ["SaaS", "software platform", "cloud"],
    "ecommerce": ["ecommerce", "e-commerce", "DTC", "Shopify"]
  }
}
```

## Example: Multi-Service Agency

Create multiple profiles with different weights to prioritize services:

```json
{
  "profiles": {
    "web-development": {
      "label": "Web Development",
      "weight": 1.0,
      "rolePatterns": { "high": ["CTO", "VP Engineering"], "medium": ["Director"], "low": ["Manager"] },
      "industries": ["saas", "ecommerce"],
      "signals": ["replatform", "redesign", "migration"]
    },
    "seo-services": {
      "label": "SEO Services",
      "weight": 0.8,
      "rolePatterns": { "high": ["CMO", "VP Marketing"], "medium": ["Director Marketing"], "low": ["Marketing Manager"] },
      "industries": ["ecommerce", "media"],
      "signals": ["organic growth", "SEO", "content strategy"]
    }
  }
}
```

Higher `weight` means that profile contributes more to a contact's Gold Score.

## Tuning Tips

After your first analysis, review the results and tell Claude what to adjust:

| Observation | What to Tell Claude |
|-------------|---------------------|
| Too many gold contacts | "Raise the gold threshold" |
| Wrong people flagged as referral partners | "Narrow the referral role patterns" |
| Buyers showing up as referrals | "Increase the buyer inversion weight" |
| Network hubs not getting credit | "Increase the network hub weight in gold score" |

Then rescore: `/network-intel rescore everything`
